## 阶段三：第一性原理分析（3.11）

### 问题本体

> 在刑事辩护案件中，律师为了形成有效辩护观点，需要在大量卷宗、证据材料、笔录、文书和法律规范中，完成"找、比、串、疑、证"这一整套认知劳动，但当前这个过程高度依赖人工、经验和体力，导致速度慢、遗漏多、质量不稳定、无法规模化。

**律师在刑事案件里，核心不是找一份材料，而是在不完整、不规整、信息量爆炸的材料里，建立一个有利于辩护的事实—证据—法律三层映射。**

### 律师真正卡住的五种阻塞

| 阻塞 | 说明 |
|-----|------|
| 材料获取后，无法快速建立全局模型 | 缺人物表、时间轴、证据链、指控结构 |
| 找得到材料，但找不到有用的矛盾 | 缺笔录前后矛盾、证人版本变化、证据链断裂识别 |
| 经验依赖太重，初级律师能力不稳定 | 缺问题模板、分析框架 |
| 输出结果难复用，劳动无法沉淀 | 成果散落在Word、微信、脑子、零碎批注 |
| 要的不是答案，而是可提交的半成品 | 缺原文出处、精确页码、可编辑结构、可导出文稿 |

### 律师真正要的不是答案

而是：
- **可追溯**
- **可核验**
- **可反驳**
- **可编辑**
- **保留原文锚点**

### 最有价值的产品方向

| 方向 | 价值 | 商业杀伤力 |
|-----|------|-----------|
| A：案件材料全局建模 | 降低启动成本 | 容易被视为"整理工具" |
| B：证据矛盾与异常发现 | 高价值发现 | ⭐ 高 |
| C：辩点发现与论证草稿 | 帮助初级律师 | 容易被质疑准确性 |
| D：可核验分析报告工作台 | 可落地交付 | ⭐ 高 |

**B + D 的组合最有商业杀伤力**

## 阶段四：致命错误警示

### 错误一：把"刑事案件分析"理解成"法律版ChatGPT"

用户真正要的不是陪聊，不是泛泛总结。

要的是：**基于卷宗，输出可核验、可修改、可提交的分析成果。**

### 错误二：把"证据查找"当成主价值点

检索本身会迅速商品化。

真正难的是：找冲突、找缺口、找不一致、找无法闭合的证据链、找可攻击点。

### 错误三：一开始就想做全流程

"先做Agent，再做MCP，再做Skill，再做专业工作流。"

**这不是产品路线，这是技术自嗨路线。**

客户只为一个具体结果付钱：哪个环节更快、更准、更少漏？

### 错误四：低估法律场景的"信任"门槛

刑事案件一旦错漏，代价极高。

产品设计核心不是"生成能力"，而是：可追溯、可核验、可反驳、可编辑、保留原文锚点。

## 阶段五：当前结论（3.14）

### 核心反思

> 我发现其实超强的检索系统不是必须的。我最大的问题就是一开始就追求完美主义，我必须得接受一开始只要满足需求就行，慢慢再优化。

### 技术路线确定

```
Claude Code + pdf-crop + Skill + 关系图谱
```

| 组件 | 作用 |
|-----|------|
| Claude Code | 底座Agent |
| pdf-crop (Paddle) | OCR、文书切分、清洗 |
| toc生成 | 目录导航（待验证必要性） |
| Skill | 案件分析任务编排 |
| 关系图谱 | 辅助检索，弥补grep局限 |

### 关系图谱的作用

**问题**：Claude Code的检索策略主要是关键词（grep），有一定局限性。

**解决**：
- 先生成关系图谱
- 图谱包含相关信息，解决关键词缺失
- 通过关系链发现相关证据

### PageIndex定位

- 有价值但不是通用解决方案
- 需要适配法律场景
- 后续再深入

### 下一步

验证核心闭环：

```
案件材料输入 → 问题驱动分析 → 结果溯源 → 结构化产出
```

## 阶段六：验证后的新问题（3.30）

### 已验证的结论

> Agent + 检索 + Skill 这条路是可行的。

核心闭环 `案件材料输入 → 预处理 → 知识图谱构建 → 案件分析 → 结构化产出` 技术上跑通了。

### 但暴露了新的结构性问题

#### 问题一：用户不应该承担前置工作

当前流程要求：预处理（OCR + 拆分 + 清洗）→ 知识图谱构建 → 案件分析。

对用户而言，分析一个案件要做这么多前置步骤是不合理的。即使用一个统筹 Skill 把预处理→图谱→分析串起来（我们选择了 OpenClaw 作为对话入口来降低 Claude Code 终端的门槛），**用户感知到的仍然是一个复杂系统，不是一个"交给它就行"的助理**。

#### 问题二：部署是死亡陷阱

给用户安装 OpenClaw 本身就够麻烦了。更致命的是预处理模块需要 Python 环境：uv、配源、装依赖……一个律师花两个小时配环境，然后还可能遇到兼容问题。**这不是产品，这是酷刑。**

#### 问题三：琐事黑洞

现实中的案件分析不是一个"分析"任务，而是无数个琐事的集合：
- 银行流水分析（PDF → 精确数字 → Excel）
- 各种证据的 PDF 是扫描件
- OCR 精度不够（银行流水尤其严重）
- 每个小需求都不可能单独做 Skill

不可能给每一件琐事都做一个 Skill，但不做的话分析效果就不好。

#### 问题四：需求假设可能有问题

身边的律师没有"案件分析"这个需求。他们实际的行为是：自己脑子里把任务拆碎，然后一步一步让 AI 做（比如"帮我分析一下这份银行流水"）。

**这意味着律师要的可能不是一个"案件分析系统"，而是一个能响应各种碎片化需求的助手。**

#### 问题五：OCR 精度是硬伤

很多证据是 PDF 扫描件，OCR 出来的数据不够精确。尤其是银行流水，要求一丝不差，还要能生成对应 Excel。传统 OCR 方案（PaddleOCR）在这个场景下精度不够。

### 关键反思

| 之前的假设 | 现实 |
|-----------|------|
| 律师需要一个"案件分析系统" | 律师更像是需要一个能响应各种碎片需求的助手 |
| 用户会接受一定的安装门槛 | 任何超过5分钟的安装都是流失点 |
| 预处理 → 图谱 → 分析的流程合理 | 用户不应该感知到这个流程 |
| 每个功能做 Skill | 琐事无穷无尽，不可能穷举 |
| OCR 能解决 PDF 问题 | 关键场景（银行流水）精度不够 |

## 阶段八：Harness Engineering——从执行者到骑手（4.2）

### 核心认知转变

> **我不是在"写 Skill"，我是在"设计让 AI 能写好 Skill 的 Harness"。**

调研了 Harness Engineering 概念（Anthropic、OpenAI Codex 团队的实践），意识到一人公司的核心杠杆不是自己加班，而是设计好 Harness 让 AI 高效执行。

### 和之前认知的关系

| 之前的认知 | 现在的修正 |
|-----------|-----------|
| Skill 是最值钱的（阶段二） | 没变，但**设计 Harness** 比写 Skill 本身更有价值 |
| 一人做不了全流程（阶段六） | 一个人+AI 协作可以——关键是分工对 |
| 每个功能要自己做 | 重新分工：我定义"好"，AI 在 Harness 内执行 |
| 技术自嗨 = 复杂架构（阶段二） | Harness 不是技术自嗨——是把律师反馈编码成约束 |

### 核心分工模型

```
我（骑手）                    AI（马）
├─ 定义"好"的标准             ├─ 按"好"的标准生成内容/代码
├─ 提供真实案例和律师评价      ├─ 用案例迭代优化
├─ 做最终判断（能用/不能用）   ├─ 根据判断修改 Harness
├─ 跟律师沟通                 ├─ 不碰律师沟通
└─ 设计 Harness               └─ 在 Harness 内执行
```

### 关键原则

1. **输出不好时，改 Harness 不改输出** — 这是 Harness 工程的核心
2. **律师反馈 = Harness 的评估标准** — 每次律师说"不对"，都是给 Harness 加约束的机会
3. **Anthropic 实践验证**：Generator + Evaluator 分离，LangChain 只改 Harness 不改模型就从 52.8% 跳到 66.5%
4. **我的核心竞争力不是写代码，是把法律专业知识编码成 AI 能执行的约束**

### 对执行的影响

所有任务重新按 Harness 模型分工——详见 [[任务看板]] 重构版。

## 阶段九：从"专业知识不足"到"推理纪律缺失"的认知修正（4.4）

### 起点

黄律和吴律都反馈AI法律分析专业水准不够。4/4讨论后，初步归因为"通用Agent缺乏法律专业知识"（推理能力 ≠ 专业知识），设计了法考真题摸底测试。

### 测试过程

1. 搜索获取法考真题（2020-2025年主观题回忆版）
2. 编写6道案例分析题（继承、合同、婚姻、侵权、劳动、物权各1道）+ 完整标准答案
3. 3个Agent并行测试（无法律专项prompt），对照标准答案比对

### 第一次诊断（测试报告初版）

| 案件类型 | 初版结论 |
|---------|---------|
| 继承纠纷 | ✅ 通过 |
| 合同纠纷 | ⚠️ 部分通过（概念精度不足） |
| 婚姻财产 | ⚠️ 部分通过（概念边界模糊） |
| 侵权纠纷 | ✅ 通过 |
| 劳动争议 | ⚠️ 部分通过（缺计算标准） |
| 物权纠纷 | ✅ 通过 |

初版报告结论：**缺的是精确法律概念，不是逻辑推理**。3/6通过、3/6部分通过。

### 关键转折：真实案例推翻了测试结果

用户把黄律的**真实继承案例**（陈继权房产继承份额）发给OpenClaw跑了一遍。

**OpenClaw第一次回答错了**：只算了陈邓名下登记的 1/3 份额，遗漏了他从父亲陈继权处继承的 1/36。

但AI**完全知道**代位继承、法定继承规则——追问后立刻修正，并给出完整正确的三层继承计算。

### 核心认知突破

**测试报告的"继承纠纷✅通过"是假象。** 测试题太简单（只有一代继承+离婚协议），Agent轻松通过。真实案件是三代人跨时间线的多层继承，Agent第一次就错了。

更深层的发现：

| 维度 | 初版诊断 | 修正诊断 |
|------|---------|---------|
| 核心问题 | **专业知识不足**——AI不知道某些法律概念 | **推理纪律不足**——AI知道规则但不系统应用 |
| 继承评定 | ✅通过 | **❌不通过**（简单题过、复杂题错） |
| 解决方向 | 注入知识 | 注入**强制步骤模板**（先铺时间线再算） |
| 方向偏差 | 如果按"知识不足"→ 建知识库、做RAG | 如果按"纪律不足"→ 写步骤约束、做流程Skill |

**这个区分的代价巨大**：如果归因为"专业知识不足"，解决方向是建知识库、做RAG，投入大且效果不确定。如果归因为"推理纪律不足"，解决方向是写强制步骤模板和流程Skill，投入小且立竿见影。

### 错误模式的三层分析

AI在陈继权继承案中的错误不是孤立的，而是暴露了**三层推理纪律缺失**：

1. **静态思维**：把"份额"当固定标签（"陈邓占1/3"），不追问"他还继承了什么？"
2. **跳过时间线**：不先铺完整死亡→继承时间线就直接跳到计算
3. **锚定效应**：被最显性的信息锁定，停止深入

**这三点和吴律反馈的"把'修改完善'理解成'填空'"是同一类问题**——不是不懂法律，是缺少"先确认理解再动手"的职业本能。

### 与之前认知的关系

| 之前的认知 | 现在的修正 |
|-----------|-----------|
| 推理能力 ≠ 专业知识（4/4上午） | 没变，但**专业知识 ≠ 解决方案**——即使知道规则也不够，关键是系统性地应用规则 |
| Harness Engineering（阶段八） | 进一步：Harness 的核心不是注入知识，是**编码推理纪律** |
| Skill教AI法律知识（阶段五） | 修正：Skill的价值不是教知识，是**把律师的隐性工作习惯编码成强制流程** |

### 对项目的影响

1. **测试方法需要升级**：每类型1道简单题不够，必须用真实复杂案例验证
2. **Skill方向收窄**：`legal-share-calc` Skill 不是教AI怎么算，而是强制"先铺时间线再算"
3. **价值主张更具体**："我的AI不会漏算你的份额"比"我的AI很懂法律"更有说服力

### 关键教训

> **诊断错误比问题本身更危险。** 归因于"知识不足"会走向建知识库；归因于"纪律不足"会走向写流程约束。方向不同，投入完全不同。用真实案例验证测试结论，不能只用标准化题目。

## 阶段十：详细诊断推翻了阶段九的过度泛化，法条库方案验证有效（4.4晚）

### 起点

阶段九从继承案得出结论："核心问题是推理纪律缺失，不是专业知识缺失"。这个判断方向正确但**过度泛化**——把一个案例的结论推广到了所有6个案件类型。

### 详细诊断过程

3个Agent并行，每个负责一个"部分通过"类型。每个Agent执行5步：设计复杂案例 → 写专家标准答案 → 模拟通用Agent分析 → 追问探测（直接问法律概念） → 诊断。

### 关键发现

| 案件类型 | 诊断结果 | 阶段九的预测 | 是否一致 |
|---------|---------|------------|---------|
| 合同纠纷 | 70%纪律 + 30%知识 | 纪律问题 | ✅ 一致 |
| 劳动争议 | 纪律缺失为主 | 纪律问题 | ✅ 一致 |
| 婚姻财产 | **70%知识 + 30%纪律** | 纪律问题 | **❌ 不一致** |

**婚姻财产和另外两个类型性质不同**：追问后Agent仍然不知道司法解释第29条（父母出资）、第78条（孳息vs增值）。这是真正的知识盲区，不是"知道但不用"。

### 阶段九犯了和它所修正的同样的错误

阶段九的核心教训是"诊断错误比问题本身更危险"。但阶段九自身也犯了过度诊断的错误：

| | 阶段九修正的错误 | 阶段十修正的错误 |
|---|---|---|
| 原诊断 | "专业知识不足"（所有类型） | "推理纪律缺失"（所有类型） |
| 修正 | 不全是知识问题，继承案是纪律问题 | 不全是纪律问题，婚姻财产是知识问题 |
| 方法 | 用一个真实案例推翻标准化测试 | 用三个并行诊断推翻单一案例推广 |

**教训的教训**：从单一案例得出的结论，不能直接推广到所有类型。每类案件需要独立诊断。

### 法条库方案与验证

基于详细诊断，设计了**MD法条库方案**（介于硬编码和RAG之间的中间路线）：

1. 创建 `3-法条库/婚姻财产纠纷-核心法条.md`（7条核心法条+判断标准+检查清单）
2. 设计prompt渐进式调用机制（按案件类型查阅，不全量注入）
3. A/B测试验证效果

**测试结果**：

| 指标 | 无法条库 | 有法条库 | 改善 |
|------|---------|---------|------|
| "不确定"标注 | 8处 | 1处 | **-87.5%** |
| 法条引用 | 0 | 每项1-2处 | 全覆盖 |
| 增值性质区分 | 完全模糊 | 精确引用78条 | 质的飞跃 |
| 父母出资定性 | 不敢断言 | 引用29条给判断流程 | 质的飞跃 |

**投入产出**：一个法条库MD（约3000字）+ 一段prompt模板（约500字）= 几小时工作，换来87.5%的不确定消除。

### 法条库对不同类型的效果预测

| 类型 | 问题性质 | 法条库预期效果 | 优先级 |
|------|---------|-------------|-------|
| 婚姻财产 | 70%知识缺失 | **显著**（已验证） | P0 |
| 合同纠纷 | 70%纪律缺失 | 中等（需配合步骤模板） | P1 |
| 劳动争议 | 纪律缺失 | 低（需配合计算检查表） | P2 |
| 继承纠纷 | 纪律缺失 | 低（需配合时间线模板） | P2 |

### 与之前认知的关系

| 之前的认知 | 现在的修正 |
|-----------|-----------|
| 核心问题是纪律不是知识（阶段九） | **不能一刀切**——合同/劳动/继承是纪律问题，婚姻财产是知识问题。需按类型分别诊断 |
| Skill教推理纪律（阶段九） | 修正：**纪律型 → 写步骤约束；知识型 → 注入法条库**。两种类型用两种工具 |
| Harness编码推理纪律（阶段八） | 进一步：Harness既编码纪律（步骤模板），也编码知识（法条库调用机制） |
| 不要建知识库做RAG（阶段六废弃类案检索） | **修正**：RAG方向整体废弃没错，但**轻量法条库MD** ≠ RAG。法条库是精选的、人工维护的、可控的，和全自动检索是两回事 |

### 关键教训

> **从单一案例推广到所有类型，和从标准化测试得出结论一样危险。** 阶段九用继承案推翻了标准化测试的"✅通过"，但随即犯了同样的错误——把继承案的"纪律问题"推广到了全部类型。每类案件需要独立诊断，才能确定是纪律问题还是知识问题。

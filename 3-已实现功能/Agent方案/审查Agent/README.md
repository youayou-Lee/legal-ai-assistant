# 多模型交叉审查Agent

## 基本信息
- **技术方案**：OpenClaw Agent 原生多模型配置（零代码，纯配置实现）
- **一句话说明**：两个AI从对方律师和法官视角审查法律文书，发现问题后自动修正
- **部署位置**：`~/.openclaw/agents/reviewer-deepseek/`、`~/.openclaw/agents/reviewer-qwen/`、`~/.openclaw/workspace-lawyer/AGENTS.md`
- **当前状态**：v2闭环协议已完成模拟评测（5/5缺陷全部发现，共识100%），待部署到律师端用真实案例验证

## 功能描述

解决的核心问题：AI生成的法律文书存在盲区（填空不调结构、跳推理步骤、金额计算错误），单模型无法自检。审查Agent通过两个不同模型从"对方律师"和"法官"两个独立视角审查同一份文书，发现共识问题后自动修正，律师只看最终版。

## 实现架构

### 三个配置文件（零代码）

| 文件 | 作用 |
|------|------|
| `reviewer-deepseek/agent/AGENTS.md` | 对方律师视角审查员 + 按文书类型的结构化检查清单 |
| `reviewer-qwen/agent/AGENTS.md` | 法官视角审查员 + 按文书类型的结构化检查清单 |
| `workspace-lawyer/AGENTS.md` | Lawyer主Agent的闭环修正协议（v1→审查→v2） |

### 闭环流程

```
律师发任务 → lawyer生成v1
  → sessions_spawn 同时调度两个审查员
    → DeepSeek: 按检查清单逐项攻击弱点
    → Qwen: 按检查清单逐项评判风险
  → lawyer整合结果（共识/单点/误报）
  → 自动修正 → 输出v2 + 审查摘要
  → 如共识问题≥3个 → 提示人工复核
```

### v2核心设计

1. **结构化检查清单**：按文书类型（起诉状/答辩状/预判结果）给专门检查清单，不是泛泛攻击
2. **闭环修正协议**：5步流程（解析→分类→修正→输出v2→升级判断），不做v3
3. **取高不取低**：两个审查员严重程度冲突时，采信更高者
4. **实务容忍度**：区分"法律缺陷"和"实务可接受表述"，控制误报

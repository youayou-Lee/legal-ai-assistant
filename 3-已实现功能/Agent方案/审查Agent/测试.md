# 审查Agent — 测试

---

## 📋 一键测试

> **Prompt拼接规则**：先跑上游Skill测试，把AI生成的输出包裹进审查指令。
>
> 模板：`请审查以下{文书类型}，找出所有法律问题：\n\n---\n{上游Skill的输出}\n---`

### 步骤

1. 从以下上游测试任选一个，复制「📋 一键测试」到 OpenClaw 执行：
   - [[../../Skill方案/起诉状生成/测试]] → 生成起诉状
   - [[../../Skill方案/答辩状生成/测试]] → 生成答辩状
   - [[../../Skill方案/预判模块/测试]] → 生成预判分析

2. 复制AI生成的完整输出（不含推理过程），拼接为：

```
请审查以下起诉状，找出所有法律问题：

---

[粘贴AI生成的起诉状全文]

---
```

3. 粘贴发送，等待两个审查员（DeepSeek对方律师 + Qwen法官）返回结果

**验证重点**：
- 总发现率 ≥ 80%（至少一方发现的真实问题 / 输出中实际存在的问题）
- 共识率 ≥ 60%（双方都发现的问题 / 总发现问题）
- 误报率 ≤ 10%

## v1测试：初步验证（3个预设缺陷）

### 案件背景
民间借贷，本金150,000元，实际到账141,000元（砍头息9,000元），已还款20,000元。

### 预设缺陷

| # | 缺陷描述 | 类型 | 严重程度 |
|---|---------|------|---------|
| D1 | 砍头息9,000元未处理，本金仍按150,000元主张 | 金额计算错误 | 🔴致命 |
| D2 | 已还款20,000元未从本金中扣除 | 金额计算错误 | 🔴致命 |
| D3 | 利息计算基数错误（应用141,000元，实际用150,000元） | 金额计算错误 | 🟡重要 |

### 测试方法
- 两个审查员独立审查，对比发现率
- 关注：交叉验证是否提升发现率

## v2闭环测试：完整验证（5个预设缺陷）

### 案件背景
民间借贷，本金50万元，月利率2%（年化24%），期限2023.5.15-2024.5.15，被告前三月付息后停付。

### 预设缺陷

| # | 缺陷描述 | 类型 | 严重程度 | 预期被谁发现 |
|---|---------|------|---------|-------------|
| D1 | 月利率2%超过LPR四倍上限（约13.8%） | 金额计算错误 | 🔴致命 | 双方共识 |
| D2 | 未引用民间借贷司法解释关于利率上限规定 | 法条遗漏 | 🟡重要 | Qwen独立 |
| D3 | "前三个月正常付息"无证据支撑 | 事实无证据 | 🔴致命 | 双方共识 |
| D4 | 未主张逾期利息（到期后至实际清偿日） | 遗漏请求 | 🔴致命 | 双方共识 |
| D5 | 引用第676条但未据此主张逾期利息 | 法条与请求不对应 | 🟡重要 | 双方共识 |

**设计特点**：v2缺陷覆盖5个不同类型（金额、法条、证据、请求、逻辑），难度远高于v1的纯金额缺陷。

### v2检查清单（新增结构化）

**DeepSeek（对方律师视角）起诉状检查清单**：
1. 诉讼请求明确性
2. 事实可证性
3. 法条对应性（含"必须引用的特别法/司法解释"子项）
4. 金额计算准确性
5. 证据覆盖度
6. 程序问题

**Qwen（法官视角）起诉状检查清单**：
1. 诉请可执行性
2. 事实逻辑链
3. 法条适用正确性
4. 请求权基础审查
5. 当事人适格性
6. 管辖合规

### 测试方法

**步骤1**：模拟DeepSeek按检查清单逐项审查
**步骤2**：模拟Qwen按检查清单逐项审查
**步骤3**：交叉验证 — 分类为共识/单点/误报
**步骤4**：闭环修正 — 按协议v2自动修正生成v2
**步骤5**：升级判断 — 共识问题≥3个时触发人工提示

### 评分标准

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| 总发现率 | 至少一方发现的缺陷/预设缺陷总数 | ≥80% |
| 共识发现率 | 双方都发现的缺陷/预设缺陷总数 | ≥60% |
| 误报率 | 误报数/总报告问题数 | ≤10% |

### 版本对比设计

v1→v2的关键差异：
- v1检查清单：泛泛的"攻击弱点"/"客观评判"
- v2检查清单：按文书类型的结构化6项检查
- v1闭环：发现问题后"告知用户"
- v2闭环：5步修正协议（解析→分类→修正→输出→升级判断）
- v2新增：取高不取低、实务容忍度

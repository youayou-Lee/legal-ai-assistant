# 庭审提纲生成实现思考

## 核心洞察

### 庭审提纲的本质是"重组"不是"生成"

庭审提纲和其他文书Skill的本质区别在于：起诉状/答辩状是从零生成的文书，而庭审提纲是对已有信息的**按庭审流程重组**。

律师在准备庭审提纲时，脑海中已经有：
- 起诉状/答辩状的内容（开庭陈述）
- 证据清单和证明目的（举证环节）
- 对方可能怎么打（质证+辩论预案）

庭审提纲的价值不在于产生新信息，而在于**确保律师在庭上每个环节都有预案、不遗漏**。

### 为什么选择"五张纸"而不是"一份文档"

调研发现律师实务中的"五张纸法则"：

1. **事实时间线**
2. **诉请/答辩要点+法条对照**
3. **举证质证要点一览**
4. **争议焦点预判+应对预案**
5. **发问提纲+法官提问预案**

选择模块化输出的理由：

1. **律师在庭上需要快速翻阅**：一份20页的文档不如5张独立的表格
2. **不同环节用不同的纸**：举证质证环节翻第3张，辩论环节翻第4张
3. **可选择性打印**：经验丰富的律师可能只需要争议焦点和发问清单
4. **实务惯例**：调研结果显示这是律师的标准做法

### 为什么独立运行而不是串联预判模块

设计时有两种方案：

**方案A（串联）**：先跑opponent-prediction → 拿到预判结果 → 输入庭审提纲Skill
**方案B（独立）**：庭审提纲Skill内部包含预判逻辑，一次完成

选择方案B的理由：

1. **减少操作步骤**：律师不需要跑两个Skill
2. **避免信息丢失**：串联过程中预判结果可能被截断或简化
3. **预判逻辑足够轻量**：庭审提纲中的预判不需要opponent-prediction的完整6步，只需要核心预判
4. **用户选择**：确认独立运行更符合律师习惯

### 争议焦点为什么限制≤3个

调研发现一个重要规律：

> "超过3个以上的争议焦点，这个焦点其实已经不是案件的'命脉'，其他的非关键性重要的争议点可以适当在质证意见中进行陈述。"

这个约束的价值：
1. **强迫律师（和AI）聚焦核心矛盾**：避免"什么都想说，什么都说不清"
2. **符合法官注意力规律**：法官在庭审中关注的焦点也是有限的
3. **提高庭审效率**：围绕核心焦点展开辩论更有说服力

## 设计决策记录

### 决策1：6步推理框架

| Step | 核心约束 | 设计意图 |
|------|---------|---------|
| 0 | 锁定立场+案件类型 | 原告/被告策略完全不同，必须先确定 |
| 1 | 提取事实时间线 | 按"五张纸"第一张：案件基础信息 |
| 2 | 诉请/答辩+法条对照 | 按"五张纸"第二张：己方主张+法律依据 |
| 3 | 举证质证要点 | 按"五张纸"第三张：双表格（己方举证+对方质证） |
| 4 | 争议焦点+预案 | 按"五张纸"第四张：≤3个核心焦点 |
| 5 | 发问+法官提问 | 按"五张纸"第五张：发问清单+法官预案 |
| 6 | 自检 | 对照9项质量标准 |

### 决策2：表格化输出

庭审提纲全部使用表格而非段落，因为：
1. 律师在庭上只有几秒钟查阅时间
2. 表格的行列结构天然适合"定位+扫描"
3. 对比性信息（如己方观点 vs 对方预判）用表格更直观

### 决策3：法言法语约束层的应用

庭审提纲遵循与其他文书Skill相同的法言法语标准。但有一个区别：庭审提纲的质证意见和发问清单可以更口语化一些（因为律师在庭上是口头表达的），这部分在v1.1中考虑调整。

## 测试中的意外发现

### 发现：利率合法性检查位置不当

测试中，Step 2的利息计算使用了约定月利率1.5%（年化18%），直到Step 4才发现超过LPR四倍（14.6%）。

这个问题揭示了Skill设计的一个盲区：**利息计算应该在计算前先验证利率合法性**，而不是算完再发现超限。

类似的问题在其他文书Skill中也可能存在——**先执行后检查**不如**先检查后执行**。

## 方法论验证

### Harness Engineering 再次验证

庭审提纲的开发过程再次验证了 Harness Engineering 的核心理念：

1. **调研先行**：通过WebSearch调研律师庭审准备的实务做法，发现了"五张纸法则"——这不是凭空设计，而是从实务中提炼
2. **用户确认方向**：模块化输出vs完整文档、独立运行vs串联预判，两个关键决策都由用户确认
3. **测试驱动**：用模拟案例测试验证5个维度的质量

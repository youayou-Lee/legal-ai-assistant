# 法院文书自动处理 — 实现思考与反思

## 核心发现

这个 Skill 的价值不在于技术复杂度（浏览器自动化本身不难），而在于**切中了一个高频痛点**：律师每天收到大量法院短信，但处理流程繁琐（打开链接→验证码→下载→记录→设置日历提醒），容易遗漏重要时限。

**AI 在这里的角色不是"生成内容"，而是"自动化操作"**——这和起诉状、答辩状等"推理类 Skill"本质不同。它更像一个 RPA（机器人流程自动化），只是用 AI 做了触发识别和信息提取。

## 设计决策

### 1. 为什么用 Skill 而不是独立脚本？

- **律师通过微信使用**：所有交互都在微信内完成，Skill 可以直接对接 OpenClaw 微信通道
- **无需额外安装**：Skill 是 OpenClaw 原生能力，不需要律师装任何东西
- **触发自然**：律师只需转发短信，不需要学习新操作

**放弃的方案**：写一个 Python 脚本定时检查短信——太重，律师不会配置。

### 2. 为什么硬编码验证码？

当前验证码是手机后六位（886288），直接写死在 Skill 里。这是因为：
- 律师的手机号不变，验证码固定
- 动态获取验证码需要额外的短信转发或监听机制，当前不具备条件
- 如果手机号变更，改 Skill 配置即可

**权衡**：牺牲了通用性，换取了实现简单和可靠性。

### 3. 为什么提醒按文书类型区分？

不同文书的时限要求不同，遗漏后果严重程度也不同：
- **传票**：错过开庭 → 缺席判决，后果最严重
- **判决书**：错过上诉期 → 判决生效，后果严重
- **应诉通知书**：错过答辩期 → 丧失答辩权
- **举证通知书**：错过举证期 → 证据不被采纳

按类型设置不同提醒时机是必要的，不能用统一规则。

## 方法论验证

这个 Skill 验证了 **Harness Engineering 中"识别律师真实工作流"的重要性**：

- 律师没有说"我需要一个法院短信处理工具"——这是从律师工作流中观察到的痛点
- 解决方案完全贴合律师现有习惯（转发短信到微信），不需要改变行为
- 和项目方向一致：**不是建平台让律师来用，而是在律师已有的工具里嵌入 AI 能力**

## 教训

### 1. YAML frontmatter 静默失败是陷阱

OpenClaw 的 SKILL.md 如果 frontmatter 格式错误（缺少闭合 `---`），不会报错，Skill 直接不加载。AI 收到匹配消息时找不到 Skill，默认用 WebSearch 回复。**表现是"AI 不用 Skill"，根因是"Skill 没被加载"**——这个诊断花了不少时间。

**教训**：新建 Skill 后，必须用 `openclaw skills list` 确认加载成功，不能假设写了就生效。以后新建 Skill 时要先用 `python3 -c "import yaml; ..."` 验证 frontmatter 格式。

### 2. 如果重来会怎么做

- 先写最小可用版本（只做识别+通知），验证律师真的需要，再做完整自动化
- 把验证码做成配置项而非硬编码，方便未来扩展到不同律师

# 功能文档撰写规范

> 2026-04-08 v2修订：Skill类功能必须包含 README + 测试（含一键测试命令）+ 调研（如有WebSearch调研），其他子文件按需扩展。目标：在别的电脑上能通过一键测试快速验证。

---

## 核心原则

1. **只写真话** — 不知道就写"未验证"，不编造测试结果或调研结论
2. **验证状态必填** — 每个功能必须写清楚：在哪个环境测过、有没有律师验证过
3. **一键可测** — 每个Skill必须有可直接复制粘贴到OpenClaw的一键测试命令
4. **简洁优先** — 能用3行说清的不用30行，不为凑字数写内容
5. **按需扩展** — README+测试必填，调研看情况，其他子文件有了真实内容再写

## README.md — 必填

每个功能**必须**有 README.md，内容必须让不了解项目的人读完就知道这个功能是做什么的。

### 必须包含的段落

```markdown
---
title: {功能中文名}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [相关标签]
---

# {功能中文名}

## 技术方案
{OpenClaw Skill / Agent配置 / Markdown知识库 / Python脚本}

## 一句话说明
{不超过30字}

## 部署位置
{文件路径}

{功能核心内容：流程、设计思路、关键改进等}

## ⚠️ 验证状态
- **测试环境**：{Claude Code模拟 / OpenClaw已部署 / 律师端已验证}
- **OpenClaw端到端运行**：✅已验证 / ❌从未运行过
- **场景覆盖**：{已测试的案件类型}
- **律师验证**：✅已确认 / ❌未验证

## 目录索引
- [[调研]] — ...
- [[测试]] — ...
- [[结果]] — ...
- [[思考]] — ...
- [[存在的问题]] — ...
- [[后续方向]] — ...
```

### 验证状态写法要求

这是最重要的段落，必须诚实：
- ✅ 表示真的跑过/验证过
- ❌ 表示没做过
- 不用模糊说法如"待验证"——要写清楚"从未在OpenClaw中运行过"

---

## 测试.md — 必填（Skill类功能）

Skill类功能**必须**有测试.md，且必须包含一键测试命令，确保在别的电脑上能快速验证。

### 必须包含的内容

```
## 📋 一键测试

> **用法**：复制下方框内全部内容，粘贴到 OpenClaw 对话框发送即可。

\`\`\`
{完整的测试输入，包含：案件事实 + 己方身份 + 诉讼请求/答辩意见 + 证据清单}
\`\`\`

**验证重点**：
1. {验证点1}
2. {验证点2}
...
```

### 完整的测试.md结构

| 段落 | 是否必填 | 内容 |
|------|---------|------|
| 📋 一键测试 | **必填** | 可直接复制粘贴的完整测试输入 + 验证重点 |
| 测试案例 | 必填 | 基本案情、证据清单、设计的风险点 |
| 验证清单 | 必填 | 按Step逐条列出检查项（checkbox格式） |
| 评分标准 | 必填 | 5个维度×权重×评分细则 |
| 测试结果 | 必填 | 已完成的测试结果汇总表（v1/v2/...） |
| 边界测试 | 推荐填写 | 2-4个边界场景的预期输出 |
| 下一步计划 | 推荐填写 | 短中长期测试计划 |

### 一键测试命令的写法要求

- **自包含**：包含所有必要信息，不需要额外准备材料
- **有风险点设计**：故意包含1-2个风险点（如砍头息、利率超限），验证AI能否发现
- **验证重点明确**：列出5-7个验证点，标注"应怎样"
- **格式**：用代码块包裹，方便整段复制

### 评分标准模板

```
| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 完整性 | 25% | 所有模块/步骤全部覆盖，无遗漏 |
| 实用性 | 25% | 输出可直接使用 |
| 精准性 | 20% | 法条正确、事实准确 |
| 对抗性 | 15% | 预判对方策略的覆盖度 |
| 格式规范 | 15% | 表格化、法言法语、格式统一 |
```

每个维度按4档评分：优秀(90%+)、良好(80%+)、及格(60%+)、不及格(<60%)

## 调研.md — 做过WebSearch时必填

如果开发过程中做了WebSearch调研，**必须**记录调研发现。

### 必须包含的内容

| 段落 | 内容 |
|------|------|
| 调研目标 | 这次调研想搞清楚什么 |
| 调研方法 | 搜索了哪些关键词、覆盖了哪些来源 |
| 调研发现 | 每个发现的详细说明+来源链接 |
| 与项目的关系 | 调研结论如何指导了Skill设计 |
| 调研结论 | 3-5条核心结论 |

### 调研发现的写法要求

- 每个发现要有**具体来源链接**
- 区分"行业共识"和"个别观点"
- 必须说明"这个发现如何影响了Skill设计"

## 按需扩展的子文件

以下子文件只在有真实内容时才创建。**不要为了凑文件数写废话。**

| 文件 | 什么时候创建 | 写什么 |
|------|-------------|--------|
| `结果.md` | 有多版本对比数据时 | v1→v2对比表、按Step的详细结果、关键发现 |
| `思考.md` | 实现过程有重要发现时 | 为什么这样设计、方法论验证、设计决策记录 |
| `存在的问题.md` | 功能有已知限制时 | 按优先级列出：已修复的（标记~~删除线~~）+待修复的 |
| `后续方向.md` | 有明确的迭代计划时 | 短/中/长期具体计划+优先级排序+依赖关系 |

### 反例（不要写的内容）

以下内容是上一轮文档的典型问题，不要重复：

- 没做过调研就写"调研.md"，内容是泛泛的行业介绍
- 没有真实测试就写详细的"测试方法"和"验证标准"
- 写"适用场景"列举一堆理想场景，但实际只测试了一个
- 写"使用指南"和"快速开始"，但功能从未真实运行过
- 画复杂的架构图描述"层次结构"，但实际就是单个SKILL.md文件

## 通用规范

### YAML Frontmatter

所有 `.md` 文件包含 YAML frontmatter：

```yaml
---
title: {文档标题}
type: {test-results / research / reflection / testing / limitations / roadmap / skill-overview}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [{标签}]
---
```

### 文档间链接

使用 Obsidian wikilink 格式：
- 同级文档：`[[测试]]`
- 跨功能引用：`[[../起诉状生成/README]]`
- 项目内引用：`[[当前状态]]`、`[[任务看板]]`

### 每个子文件末尾

添加"相关文档"段落，链接到同目录其他文件：

```markdown
## 相关文档
- [[README]] — 功能总览
- [[测试]] — 一键测试+验证方法
- [[结果]] — 测试结果
```

---

## 检查清单

新建Skill功能文档时：

- [ ] README.md 存在且包含验证状态段落
- [ ] 测试.md 存在且包含一键测试命令（可直接复制到OpenClaw）
- [ ] 调研.md 存在（如果做过WebSearch调研）
- [ ] 验证状态每个 ✅/❌ 都反映真实情况
- [ ] 没有编造的测试结果或调研结论
- [ ] 内容基于实际做过的事，不是理想化的描述
- [ ] 子文件只在有真实内容时才创建

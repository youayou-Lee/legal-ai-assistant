# 功能文档撰写规范

> 2026-04-08 修订：从"7文件必填"改为"README必填+按需扩展"，避免为凑文件数写废话。

---

## 核心原则

1. **只写真话** — 不知道就写"未验证"，不编造测试结果或调研结论
2. **验证状态必填** — 每个功能必须写清楚：在哪个环境测过、有没有律师验证过
3. **简洁优先** — 能用3行说清的不用30行，不为凑字数写内容
4. **按需扩展** — 简单功能只需 README，复杂功能再补充子文件

---

## 目录结构

```
3-已实现功能/
├── Skill方案/          ← OpenClaw Skill 实现
│   └── {功能名}/
│       └── README.md   ← 必填（唯一必填文件）
│       └── *.md        ← 按需添加
├── Agent方案/          ← OpenClaw Agent 实现
├── 知识基座方案/       ← 知识资源
├── 通用工具/           ← 跨功能工具
└── _已归档/            ← 已废弃/搁置的功能
```

### 命名规则

| 项目 | 规则 | 示例 |
|------|------|------|
| 方案组目录 | `{技术方案}方案` | `Skill方案/` |
| 功能目录 | 中文名称 | `答辩状生成/` |
| 文档文件 | 角度名称.md | `测试.md` |

## README.md — 唯一必填文件

每个功能**必须**有 README.md，内容必须让不了解项目的人读完就知道这个功能是做什么的。

### 必须包含的段落

```markdown
---
title: {功能中文名}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [相关标签]
---

# {功能中文名}

## 技术方案
{OpenClaw Skill / Agent配置 / Markdown知识库 / Python脚本}

## 一句话说明
{不超过30字}

## 部署位置
{文件路径}

{功能核心内容：流程、设计思路、关键改进等}

## ⚠️ 验证状态
- **测试环境**：{Claude Code模拟 / OpenClaw已部署 / 律师端已验证}
- **OpenClaw端到端运行**：✅已验证 / ❌从未运行过
- **场景覆盖**：{已测试的案件类型}
- **律师验证**：✅已确认 / ❌未验证

## 目录索引（如有子文件）
- [[测试]] — ...
- [[思考]] — ...
```

### 验证状态写法要求

这是最重要的段落，必须诚实：
- ✅ 表示真的跑过/验证过
- ❌ 表示没做过
- 不用模糊说法如"待验证"——要写清楚"从未在OpenClaw中运行过"

---

## 按需扩展的子文件

只有当功能足够复杂、有真实内容可写时才创建子文件。**不要为了凑文件数写废话。**

| 文件 | 什么时候创建 | 写什么 |
|------|-------------|--------|
| `测试.md` | 有真实测试案例时 | 测试输入+验证点+实际结果 |
| `结果.md` | 有多版本对比数据时 | v1→v2对比表、关键发现 |
| `思考.md` | 实现过程有重要发现时 | 为什么这样设计、踩过的坑 |
| `调研.md` | 做过真实调研时 | 搜索到的关键信息、参考来源 |
| `存在的问题.md` | 功能有已知限制时 | 按优先级列出具体问题 |
| `后续方向.md` | 有明确的迭代计划时 | 短/中/长期具体计划 |

### 反例（不要写的内容）

以下内容是上一轮文档的典型问题，不要重复：

- 没做过调研就写"调研.md"，内容是泛泛的行业介绍
- 没有真实测试就写详细的"测试方法"和"验证标准"
- 写"适用场景"列举一堆理想场景，但实际只测试了一个
- 写"使用指南"和"快速开始"，但功能从未真实运行过
- 画复杂的架构图描述"层次结构"，但实际就是单个SKILL.md文件

## 通用规范

### YAML Frontmatter

所有 `.md` 文件包含 YAML frontmatter：

```yaml
---
title: {文档标题}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [{标签}]
---
```

### 文档间链接

使用 Obsidian wikilink 格式：
- 同级文档：`[[测试]]`
- 跨功能引用：`[[../起诉状生成/README]]`
- 项目内引用：`[[当前状态]]`、`[[任务看板]]`

---

## 检查清单

新建功能文档时：

- [ ] README.md 存在且包含验证状态段落
- [ ] 验证状态每个 ✅/❌ 都反映真实情况
- [ ] 没有编造的测试结果或调研结论
- [ ] 内容基于实际做过的事，不是理想化的描述
- [ ] 子文件只在有真实内容时才创建

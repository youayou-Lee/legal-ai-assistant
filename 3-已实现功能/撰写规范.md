# 功能文档撰写规范

> 本规范定义了「3-已实现功能」目录下每个功能模块的文档结构和内容要求。
> 新功能开发完成后，必须按此规范创建文档。

---

## 一、目录结构

每个功能创建一个独立目录，放在对应的方案组下：

```
3-已实现功能/
├── Skill方案/          ← OpenClaw Skill 实现
│   └── {功能名}/
├── Agent方案/          ← OpenClaw Agent 实现
│   └── {功能名}/
├── 知识基座方案/       ← 知识资源（法条库、案例库等）
│   └── {资源名}/
├── {新方案}/           ← 未来的新实现方案
│   └── {功能名}/
└── 撰写规范.md         ← 本文件
```

### 命名规则

| 项目 | 规则 | 正确示例 | 错误示例 |
|------|------|---------|---------|
| 方案组目录 | `{技术方案}方案` | `Skill方案/` | `skill方案/`、`skill/` |
| 功能目录 | 功能的中文名称 | `答辩状生成/` | `defense-statement/` |
| 文档文件 | 角度名称.md | `测试.md` | `test.md`、`测试结果.md` |

---

## 二、必需文件清单（7个）

每个功能目录**必须**包含以下 7 个文件：

| 序号 | 文件名 | 用途 | 必须程度 |
|------|--------|------|---------|
| 1 | `README.md` | 功能总览和导航入口 | **必须** |
| 2 | `测试.md` | 测试案例、测试方法、测试设计 | **必须** |
| 3 | `结果.md` | 测试结果、版本迭代对比 | **必须** |
| 4 | `思考.md` | 实现过程中的思考、反思、方法论验证 | **必须** |
| 5 | `调研.md` | 背景研究、行业标准、竞品分析 | **必须** |
| 6 | `存在的问题.md` | 当前局限、已知问题、待解决项 | **必须** |
| 7 | `后续方向.md` | 短/中/长期优化计划 | **必须** |

---

## 三、各文件内容规范

### 3.1 README.md — 功能总览

**作用**：打开这个目录的第一眼就明白这个功能是什么、怎么做的、做到什么程度。

**必须包含**：

```markdown
---
title: {功能中文名}
type: {Skill实现|Agent实现|知识基座|...}
version: {当前版本}
status: {规划中|开发中|已测试|已部署|已废弃}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [相关标签]
---

# {功能中文名}

## 基本信息
- **技术方案**：{OpenClaw Skill / Agent配置 / Markdown知识库 / ...}
- **一句话说明**：{不超过30字}
- **部署位置**：{文件路径或"未部署"}
- **当前状态**：{具体说明}

## 功能描述
{这个功能解决什么问题，为谁解决，解决到什么程度}

## 实现架构
{怎么实现的，关键设计决策}

## 文档索引
- [[测试]] — 测试案例与方法
- [[结果]] — 测试结果与版本对比
- [[思考]] — 实现思考与反思
- [[调研]] — 背景研究与行业标准
- [[存在的问题]] — 当前局限与已知问题
- [[后续方向]] — 优化与扩展计划
```

**质量要求**：
- 一个不了解项目的人读完 README 后能说清楚这个功能是做什么的
- 技术方案写清楚用了什么技术、怎么部署的
- 状态写清楚具体进展（不能只写"已完成"，要写"v2测试通过，待真实案例验证"）

---

### 3.2 测试.md — 测试案例与方法

**作用**：记录你是怎么验证这个功能的，让其他人能复现测试。

**必须包含**：

```markdown
---
title: {功能名} - 测试
type: 测试文档
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [测试, {功能标签}]
---

# {功能名} — 测试

---

## 📋 一键测试

> **用法**：复制下方框内全部内容，粘贴到 OpenClaw 对话框发送即可。

​```
{一段完整的prompt，包含：触发指令 + 完整案件数据 + 具体要求}
​```

**验证重点**：
1. {验证点1}（具体到可判断对错的程度）
2. {验证点2}
3. ...

---

## 测试案例：{案件类型}

### 案件背景
{案件类型、当事人、关键事实}

### 输入材料
{提供给AI的原始材料清单}

## 测试方法
{用什么方法测试的，关注什么指标}

### 验证点清单
{逐项列出需要验证的点}

### 评分标准（如适用）
{各维度的评分规则}

## 版本对比设计
{如果有多版本测试，说明v1和v2的测试差异}
```

**「📋 一键测试」撰写要求**：

| 要求 | 说明 | 正确 | 错误 |
|------|------|------|------|
| 自包含 | 框内包含所有案件数据，不需要外部参考 | 框内有完整的当事人、金额、日期、证据 | "案件数据见下方详细案例" |
| 自然语言 | 用律师会说的话，不要技术术语 | "帮我写起诉状，案件如下：..." | "触发civil-complaint Skill，参数如下..." |
| 一键触发 | 复制粘贴即触发，不需要额外操作 | 粘贴发送即可 | 需要先修改占位符再发送 |
| 独立验证 | 验证重点写在框外，框内只有输入 | 框内无验证信息 | 框内包含"检查是否..." |

**上下流Skill引用**（如审查Agent）：

如果测试输入依赖上游Skill的输出，不要重新造数据，直接引用：

```markdown
## 📋 测试流程：上游输出 → 审查输入

### 第1步：先跑上游测试
- 去 [[../上游Skill名/测试]] 的「📋 一键测试」区域
- 复制prompt到OpenClaw，等AI生成结果
- **复制AI生成的完整输出**

### 第2步：粘贴触发本Skill
​```
{触发指令}

---

[粘贴第1步AI生成的完整输出]

---
​```
```

**质量要求**：
- 测试案例要足够复杂，能暴露真实问题（不能用简单案例产生假阳性）
- 验证点必须具体可检查（不能写"输出质量好"，要写"利息计算按561条先息后本"）
- 测试案例中的数据要具体到金额、日期、人名
- 「📋 一键测试」必须放在文件最前面，紧接标题之后

### 3.3 结果.md — 测试结果

**作用**：如实记录测试结果，好坏都写，版本迭代要有对比。

**必须包含**：

```markdown
---
title: {功能名} - 结果
type: 测试结果
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [结果, {功能标签}]
---

# {功能名} — 测试结果

## 版本迭代对比

### {v1}结果
{具体数据，不能只写"通过/不通过"}

### {v2}结果
{具体数据}

### 版本对比表
| 维度 | v1 | v2 | 改进原因 |
|------|----|----|---------|
| ... | ... | ... | ... |

## 关键发现
{测试中发现的最重要的结论}

## 仍存在的问题
{测试中暴露但未解决的问题}
```

**质量要求**：
- 数字必须具体（不能写"本金计算改善了"，要写"剩余本金从700,000元修正为749,713.34元"）
- 版本对比表必须量化
- 好的结果和差的结果都要记录，不能只写好的

---

### 3.4 思考.md — 实现思考与反思

**作用**：记录实现过程中的思考、判断和取舍。这是最重要的文档——它记录了"为什么这样做"。

**必须包含**：

```markdown
---
title: {功能名} - 思考
type: 反思文档
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [思考, 反思, {功能标签}]
---

# {功能名} — 实现思考与反思

## 核心发现
{实现过程中最重要的认知收获}

## 设计决策
{做了什么选择，为什么这样选，放弃了什么}

## 方法论验证
{这个功能的实现验证或挑战了什么方法论}

## 教训
{踩过的坑，下次要避免的}
```

**质量要求**：
- 必须写"为什么"，不能只写"做了什么"
- 必须包含至少一个"教训"或"如果重来会怎么做"
- 与项目方法论（Harness Engineering、推理纪律假说等）的关联

---

**作用**：记录开发前的背景研究，包括行业标准、法律依据、竞品分析等。

**必须包含**：

```markdown
---
title: {功能名} - 调研
type: 调研文档
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [调研, {功能标签}]
---

# {功能名} — 背景研究

## 问题背景
{为什么需要这个功能，痛点是什么}

## 行业标准
{法律/行业对这个领域的标准要求}

## 法律依据（如适用）
{相关法条、司法解释、示范文本}

## 竞品/对标分析（如适用）
{同类产品或服务是怎么做的}

{参考了哪些资料，链接到哪里}
```

**质量要求**：
- 法律依据要具体到法条编号
- 如果引用了项目内的研究文件，用 `[[]]` 链接（如 `[[研究背景-起诉状答辩状标准]]`）
- 调研结论要说明对功能设计的影响

### 3.6 存在的问题.md

**作用**：诚实记录当前功能的局限和问题。不写这个 = 自欺欺人。

**必须包含**：

```markdown
---
title: {功能名} - 存在的问题
type: 问题记录
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [问题, {功能标签}]
---

# {功能名} — 存在的问题

## 问题清单

### P0 — 阻塞性问题
{影响核心功能可用性的问题}

### P1 — 重要问题
{影响功能质量但不阻塞使用的问题}

### P2 — 一般问题
{体验或效率问题}

### P3 — 低优先级
{锦上添花的改进项}

**质量要求**：
- 每个问题必须具体（不能写"效果不好"，要写"仅测试了民间借贷一种案件类型，其他类型未验证"）
- 必须有优先级排序
- 问题不能回避，必须诚实记录

**作用**：规划接下来的优化路线，避免做完了就不管了。

**必须包含**：

---

- 必须包含与其他功能的联动计划
- 后续方向要和「存在的问题」对应（问题里的 P0 应该在短期方向里）

## 四、通用规范

### 4.1 YAML Frontmatter

所有 `.md` 文件**必须**包含 YAML frontmatter：

```yaml
---
title: {文档标题}
type: {功能实现|测试文档|测试结果|反思文档|调研文档|问题记录|规划文档|规范文档}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [{标签1}, {标签2}]
---
```

### 4.2 文档间链接

使用 Obsidian wikilink 格式建立文档关联：

- README 中链接所有同级文档：`[[测试]]`、`[[结果]]` 等
- 文档间交叉引用：`详见 [[思考]] 中的方法论验证`
- 跨功能引用：`与 [[../起诉状生成/思考]] 的发现一致`
- 引用项目内研究文件：`参考 **研究背景-起诉状答辩状标准**`

### 4.3 内容密度要求

| 指标 | 最低标准 |
|------|---------|
| README.md | ≥ 2KB |
| 测试.md | ≥ 3KB |
| 结果.md | ≥ 3KB，必须包含数据表格 |
| 思考.md | ≥ 2KB，必须包含"为什么" |
| 调研.md | ≥ 2KB，必须引用具体来源 |
| 存在的问题.md | ≥ 1KB，至少 3 个具体问题 |
| 后续方向.md | ≥ 1KB，分短/中/长期 |

### 4.4 版本迭代记录

当功能有版本更新时：
1. 在 `结果.md` 中增加新版本的结果段落
2. 更新 `README.md` 的 version 和 status 字段
3. 在 `思考.md` 中补充迭代过程中的新发现
4. 更新 `存在的问题.md`（已解决的问题删除，新发现的问题添加）

### 4.5 新方案组的创建

当出现新的实现方案（非 Skill、非 Agent、非知识基座）时：
1. 在 `3-已实现功能/` 下创建新的方案组目录
2. 方案组目录名格式：`{技术方案}方案/`
3. 新方案组下的功能目录同样遵循本规范的 7 文件结构

---

## 五、检查清单

新建功能文档时，逐项检查：

- [ ] 目录放在正确的方案组下（Skill方案/Agent方案/知识基座方案/...）
- [ ] 目录名使用中文功能名
- [ ] 7 个必需文件全部创建
- [ ] 每个文件都有 YAML frontmatter
- [ ] README.md 有 wikilink 索引
- [ ] 测试.md 文件最前面有「📋 一键测试」区域（可复制粘贴到OpenClaw直接触发）
- [ ] 一键测试框内数据自包含、自然语言、不需要额外操作
- [ ] 结果.md 有量化的版本对比表
- [ ] 思考.md 写了"为什么这样设计"
- [ ] 调研.md 引用了具体的法条/来源/项目内文件
- [ ] 存在的问题.md 至少 3 个具体问题，有优先级
- [ ] 后续方向.md 分短/中/长期，与存在的问题对应
- [ ] 每个文件内容达到最低密度标准

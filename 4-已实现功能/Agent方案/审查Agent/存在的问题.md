# 审查Agent — 存在的问题

## P0 — 阻塞性问题

### 问题1：模拟测试≠真实AI执行
- v2测试是人工模拟审查过程，真实AI模型可能不严格遵循检查清单
- 修正后的金额重算（如利息12万→6.9万）是否正确未经验证
- **待做**：用真实DeepSeek+Qwen跑完整闭环

### 问题2：仅测试了民间借贷起诉状一种输入
- 答辩状和预判结果的检查清单已设计但未测试
- 其他案件类型（买卖合同、租赁合同等）完全未覆盖
- **待做**：至少测试答辩状+另一种案件类型

## P1 — 重要问题

### 问题3：检查清单长度可能影响模型遵循率
- DeepSeek起诉状清单：6项基础 + 实务容忍度说明
- 每项还有2-4个子项
- 模型可能跳过某些子项
- **待做**：真实AI测试时观察是否逐项执行

### 问题4：闭环修正未真实执行验证
- 自动修正协议写了5步，但未用真实AI跑过
- 不确定AI修正时是否会引入新问题
- 不确定审查摘要格式AI是否能正确输出
- **待做**：用OpenClaw跑一次完整闭环

### 问题5：严重程度分级缺乏数据校准
- 🔴/🟡/🟢分级是定性判断
- v2测试中D3出现🔴vs🟢冲突（靠取高规则解决）
- 律师可能不认可AI的严重程度判断
- **待做**：收集律师对问题严重程度的反馈

## P2 — 一般问题

### 问题6：审查结果未持久化
- 审查结果仅在运行时显示，未保存到案件档案
- 无法追溯历史审查记录
- **待做**：设计审查结果持久化格式

### 问题7：不支持增量审查
- 每次审查都是全量审查，文书修改后需要重新审查全部内容
- 修改迭代效率低
- **待做**：设计增量审查方案

### 问题8：仅支持中文民事案件
- 检查清单针对中国民事法律设计
- 刑事案件审查逻辑完全不同
- 行政诉讼未覆盖
- **待做**：刑事案件审查需单独设计检查清单

## P3 — 低优先级

### 问题9：多模型调用成本和延迟
- 每次审查调用2个模型，成本是单模型的约2倍
- 并行调度（sessions_spawn）可缓解延迟，但总token消耗不变
- 长文书审查成本可能较高

### 问题10：模型依赖风险
- 依赖DeepSeek和Qwen的API可用性
- 模型升级可能导致审查风格变化
- 无备选模型方案

# 审查Agent — 背景研究

## 问题背景

吴律反馈AI"修改完善"只做了填空，没有真正审查文书质量。黄律反馈AI跳过推理步骤。根本问题：**单模型无法发现自己的盲区**。

## 行业标准：法律文书审查维度

### 起诉状审查要点（综合律师实务）

来源：[德恒律所 - 律师之道](https://www.deheheng.com/yjy/lssd/28505.html)、[iCourt - 法律文书写作方法论](https://www.icourt.cc/prac-article/755.html)、[无讼 - 必备法律文书材料清单](https://victory.itslaw.com/html/detail/74169b65-4231-48ce-93df-ae022a4e7fae/)

| 审查维度 | 审查要点 |
|---------|---------|
| 形式审查 | 当事人信息完整准确、文书格式合规、签名盖章齐全 |
| 诉讼请求 | 请求明确具体可执行、无遗漏请求 |
| 事实与理由 | 事实陈述清晰准确、逻辑链完整、证据链覆盖 |
| 法条适用 | 法条现行有效、与请求对应、未遗漏特别法 |
| 程序审查 | 管辖正确、时效未过、当事人适格 |

### 答辩状审查要点（与起诉状的差异）

来源：[知乎 - 民事答辩状基础写作要点](https://zhuanlan.zhihu.com/p/666529590)

- 必须逐项回应原告每项诉讼请求
- 未回应的请求视为自认
- 需检查自认风险（言多必失）
- 需检查程序抗辩机会（管辖异议、时效抗辩）

## 多模型协作模式调研

来源：[L-MARS论文](https://arxiv.org/html/2509.00761v3)、[Self-Correcting AI Agents](https://www.linkedin.com/pulse/building-self-correcting-ai-agents-atharva-joshi-1bcto)

### 评估的4种模式

| 模式 | 原理 | 是否采用 | 原因 |
|------|------|---------|------|
| 辩论式 | 模型互相对抗辩论 | ❌ | 审查需要明确结论，不适合无休止争论 |
| 投票式 | 多模型投票决定 | ❌ | 需要详细理由，投票无法提供推理过程 |
| 流水线式 | 串联工作 | ❌ | 审查需要互相验证，而非分工 |
| **双视角独立审查** | 独立审查+综合判断 | ✅ | 互补性强、可信度高、成本可控 |

### 关键参考：Solver → Critic → Judge 模式

业界成熟的自纠错模式：
1. **Solver**（生成）→ 产出初始方案
2. **Critic**（攻击）→ 发现问题
3. **Judge**（评判）→ 综合判断

审查Agent的设计与此对应：
- Lawyer Agent = Solver
- DeepSeek审查员 = Critic
- Qwen审查员 = Judge

## OpenClaw原生多模型能力

技术基础：
- 每个Agent可用不同模型提供商（deepseek / openrouter）
- 配置驱动，零代码
- 支持 `sessions_spawn` 异步调度子Agent

本方案完全基于OpenClaw原生能力，不引入额外代码。参考 **项目记忆：OpenClaw配置路径全景**。

## 角色设计依据

### 为什么是"对方律师 + 法官"而不是其他组合？

律师的真实思维模式：
1. "对方律师会怎么攻击我？" → 对应DeepSeek角色
2. "法官会怎么判？" → 对应Qwen角色
3. 这两个问题覆盖了律师90%的审查关切

### 模型选择

| 角色 | 模型 | 选择理由 |
|------|------|---------|
| 对方律师 | DeepSeek Chat | 逻辑严密、攻击性强、成本低 |
| 法官 | Qwen3.6 Plus | 推理深入（60% reasoning tokens）、视角全面、成本低 |

不选GPT-4/Claude：成本高、国内访问不稳定。

## 调研结论对设计的影响

1. 行业审查标准 → 转化为结构化检查清单的6个维度
2. 双视角独立审查模式 → 确定架构方案
3. Solver→Critic→Judge模式 → 验证三角色分工的合理性
4. 答辩状与起诉状的审查差异 → 确认需要按文书类型分检查清单
